# DataMonitor — 수동 테스트 시나리오

이 문서는 DataMonitor PoC의 수동 기능 검증을 위한 테스트 시나리오를 정의합니다.  
자동화 테스트 프레임워크를 사용하지 않으며, 직접 실행 후 결과를 확인합니다.

---

## 테스트 환경 준비

- DataMonitor 빌드 완료 상태
- 프로젝트 루트의 `data/` 디렉토리 접근 가능
- 각 테스트 시작 전 데이터 파일 상태를 초기화

---

## TC-01. 주문 상태별 집계 검증

### 목적

`orders.json`에 저장된 주문 데이터가 상태별로 정확히 집계되는지 확인한다.

### 사전 조건

`data/orders.json`에 아래 내용을 저장한다.

```json
{
  "nextId": 8,
  "items": [
    {"id": 1, "productName": "제품A", "quantity": 10, "status": "접수"},
    {"id": 2, "productName": "제품B", "quantity": 5,  "status": "접수"},
    {"id": 3, "productName": "제품C", "quantity": 3,  "status": "처리 중"},
    {"id": 4, "productName": "제품D", "quantity": 7,  "status": "완료"},
    {"id": 5, "productName": "제품E", "quantity": 2,  "status": "완료"},
    {"id": 6, "productName": "제품F", "quantity": 1,  "status": "완료"},
    {"id": 7, "productName": "제품G", "quantity": 4,  "status": "취소"}
  ]
}
```

### 실행 순서

1. 프로그램 실행
2. 메인 메뉴 → `4. 모니터링` 선택
3. 서브메뉴 → `2. 주문 상태별 집계` 선택

### 기대 결과

| 항목 | 기대값 |
|---|---|
| 접수 | 2 건 |
| 처리 중 | 1 건 |
| 완료 | 3 건 |
| 취소 | 1 건 |
| 합계 | 7 건 |

### 판정 기준

- 모든 건수가 기대값과 일치하면 **PASS**
- 하나라도 불일치하면 **FAIL**

## TC-02. 시료 재고 현황 검증

### 목적

`samples.json`에 저장된 시료 목록이 정확히 표시되는지 확인한다.

### 사전 조건

`data/samples.json`에 아래 내용을 저장한다.

```json
{
  "nextId": 4,
  "items": [
    {"id": 1, "name": "시료-알파", "description": "1차 채취 샘플"},
    {"id": 2, "name": "시료-베타", "description": "2차 채취 샘플"},
    {"id": 3, "name": "시료-감마", "description": "3차 채취 샘플"}
  ]
}
```

### 실행 순서

1. 프로그램 실행
2. 메인 메뉴 → `4. 모니터링` 선택
3. 서브메뉴 → `3. 시료 재고 현황` 선택

### 기대 결과

| ID | 이름 | 설명 |
|---|---|---|
| 1 | 시료-알파 | 1차 채취 샘플 |
| 2 | 시료-베타 | 2차 채취 샘플 |
| 3 | 시료-감마 | 3차 채취 샘플 |

- 화면 하단에 "총 3건" 표시

### 판정 기준

- 3개 항목이 순서대로 표시되고, 총 건수가 3이면 **PASS**
- 항목 누락, 순서 불일치, 건수 오류 시 **FAIL**

## TC-03. 전체 Dashboard 통합 검증

### 목적

Dashboard 화면이 주문 집계와 시료 목록을 동시에 정확히 표시하는지 확인한다.

### 사전 조건

- TC-01과 동일한 `orders.json` 사용
- TC-02와 동일한 `samples.json` 사용

### 실행 순서

1. 프로그램 실행
2. 메인 메뉴 → `4. 모니터링` 선택
3. 서브메뉴 → `1. 전체 Dashboard` 선택

### 기대 결과

- 주문 현황 영역: TC-01의 집계 결과와 동일
- 시료 재고 현황 영역: TC-02의 목록과 동일
- 두 영역이 구분선으로 분리되어 표시됨

### 판정 기준

- 두 영역 모두 기대값과 일치하면 **PASS**
- 어느 한 영역이라도 불일치하면 **FAIL**

## TC-04. orders.json 파일 누락 테스트

### 목적

`orders.json`이 없을 때 프로그램이 안전하게 동작하는지 확인한다.

### 사전 조건

- `data/orders.json` 파일을 삭제하거나 이름을 변경한다.
- `data/samples.json`은 정상 상태로 유지한다.

### 실행 순서

1. 프로그램 실행
2. 메인 메뉴 → `4. 모니터링` 선택
3. 서브메뉴 → `2. 주문 상태별 집계` 선택

### 기대 결과

- 프로그램이 종료되지 않는다.
- 모든 상태 건수가 0으로 표시된다.
- 합계가 0으로 표시된다.
- `data/orders.json`이 자동으로 생성된다 (빈 초기 파일).

### 판정 기준

- 프로그램이 정상 동작하고, 0건 표시 후 메뉴로 복귀하면 **PASS**
- 프로그램 비정상 종료(crash) 또는 오류 없이 잘못된 값 표시 시 **FAIL**

## TC-06. 손상된 orders.json 테스트

### 목적

`orders.json`의 내용이 올바른 JSON 형식이 아닐 때 안전하게 처리되는지 확인한다.

### 사전 조건

`data/orders.json`에 아래와 같이 손상된 내용을 저장한다.

```
{ invalid json content !!!
  "nextId": abc,
  "items": [
```

### 실행 순서

1. 프로그램 실행
2. 메인 메뉴 → `4. 모니터링` 선택
3. 서브메뉴 → `2. 주문 상태별 집계` 선택

### 기대 결과

- 프로그램이 종료되지 않는다.
- 손상된 파일이 `orders.json.bak`으로 자동 백업된다.
- 주문 건수가 모두 0으로 표시된다.

### 판정 기준

- `.bak` 파일 생성 확인 + 0건 표시 + 메뉴 복귀 시 **PASS**
- 프로그램 비정상 종료 또는 `.bak` 미생성 시 **FAIL**

## TC-08. 잘못된 메뉴 입력 테스트

### 목적

모니터링 서브메뉴에서 유효하지 않은 입력 시 오류 처리가 올바른지 확인한다.

### 실행 순서

1. 프로그램 실행
2. 메인 메뉴 → `4. 모니터링` 선택
3. 서브메뉴에서 `9` 입력
4. 서브메뉴에서 `-1` 입력
5. 서브메뉴에서 문자 `abc` 입력

### 기대 결과

- 각 케이스에서 "잘못된 입력입니다." 메시지가 출력된다.
- 프로그램이 종료되지 않고 서브메뉴가 재표시된다.

### 판정 기준

- 모든 케이스에서 오류 메시지 출력 후 메뉴 재표시되면 **PASS**
- 프로그램 종료 또는 무한 루프 발생 시 **FAIL**
